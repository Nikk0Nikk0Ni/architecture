# ДЗ 01 – Документирование архитектуры в Structurizr

**Вариант 4 – Сайт заказа услуг (аналог https://profi.ru/)**

---

## Описание системы

Платформа заказа услуг предоставляет пользователям возможность находить нужных специалистов, а специалистам — размещать свои услуги. Клиенты могут формировать заказы из выбранных услуг и отслеживать их статус.

**Основные сущности системы:**
* **Пользователь** — зарегистрированный участник платформы (может выступать клиентом или специалистом).
* **Услуга** — предложение от специалиста с описанием выполняемой работы.
* **Заказ** — сформированная заявка клиента, содержащая одну или несколько услуг.

Пользователи взаимодействуют с системой через веб-интерфейс (SPA). Запросы поступают через API Gateway и далее маршрутизируются в специализированные микросервисы, которые обрабатывают бизнес-логику и взаимодействуют с базой данных.

**Основные подсистемы:**
* **Web Application** (Клиентское приложение)
* **API Gateway** (Единая точка входа и маршрутизация запросов)
* **User Service** (Микросервис управления пользователями)
* **Service Catalog** (Микросервис каталога услуг)
* **Order Service** (Микросервис управления заказами)
* **Database** (База данных YDB для хранения информации)

**Интеграция с внешними сервисами:**
* **Email Сервис** для отправки уведомлений о статусах заказов.
* **Платежная система** для проведения оплаты за выполненные услуги.

---

## Основные API системы

В соответствии с заданием реализованы следующие операции (API):

### Пользователи:
* Создание нового пользователя
* Поиск пользователя по логину
* Поиск пользователя по маске имени и фамилии

### Услуги:
* Создание услуги
* Получение списка услуг

### Заказы:
* Добавление услуг в заказ
* Получение заказа для пользователя

---

## Роли пользователей

В системе предусмотрены следующие роли:

* **Клиент:** Ищет необходимые услуги в каталоге, добавляет услуги в заказ, просматривает историю и статусы своих заказов.
* **Специалист:** Создает новые услуги для отображения в каталоге, управляет предоставлением своих услуг.

---

## Внешние системы

* **Email Сервис (Notification Provider):** Внешний сервис для отправки транзакционных писем (уведомления об успешном формировании заказа или изменении его статуса). Протокол взаимодействия: SMTP / REST.
* **Платежная система (Payment Gateway):** Внешний сервис для безопасной оплаты заказов. Протокол взаимодействия: HTTPS / REST.

## Архитектура системы

Архитектура системы описана в стиле **C4 Model** с использованием **Structurizr DSL**. Описание архитектуры находится в файле `workspace.dsl`.

### Диаграммы

**C1 — System Context**
Диаграмма контекста системы показывает:
* Роли пользователей (Клиент, Специалист).
* Взаимодействие пользователей с платформой заказа услуг.
* Интеграцию платформы с внешними системами (Email Сервис, Платежная система).

**C2 — Container Diagram**
Диаграмма контейнеров раскрывает внутреннее устройство платформы:
* **Web Application** — пользовательский интерфейс (React / TypeScript).
* **API Gateway** — маршрутизация запросов к микросервисам (Nginx / Traefik).
* **User Service** — обработка логики пользователей (Python / FastAPI).
* **Service Catalog** — управление услугами (Python / FastAPI).
* **Order Service** — управление заказами (Python / FastAPI).
* **Database** — хранилище данных (YDB).
* Отображены протоколы взаимодействия (HTTPS/JSON, REST, SQL).

**C3 — Dynamic Diagram**
Динамическая диаграмма показывает архитектурно значимый сценарий: **Добавление услуги в заказ**. 
Сценарий включает следующие логические шаги:
* Клиент нажимает кнопку добавления услуги в интерфейсе.
* Web Application отправляет запрос на бэк (API Gateway).
* API Gateway перенаправляет запрос в Order Service.
* Order Service сохраняет данные о заказе в базе данных.
* База данных подтверждает успешную запись.
* Отправляется событие во внешний Email Сервис для уведомления.
* Успешный ответ возвращается клиенту, и интерфейс обновляется.

# ДЗ 02: REST API сервис

Архитектура платформы для заказа услуг (Микросервисы: User Service, Service Catalog, Order Service).
Реализовано на **Python + FastAPI** с in-memory хранилищем и JWT-аутентификацией.

## Запуск проекта

### Вариант 1: Через Docker 
1. Выполните команду в корне проекта:
   docker-compose up -d --build
2. API и документация будут доступны по адресу: http://localhost:8000/docs

### Вариант 2: Локально
1. Создайте виртуальное окружение: python -m venv venv
2. Активируйте его (Windows: venv\Scripts\activate, Mac/Linux: source venv/bin/activate)
3. Установите зависимости: pip install -r requirements.txt
4. Запустите сервер: uvicorn main:app --reload
5. Откройте в браузере: http://127.0.0.1:8000/docs
